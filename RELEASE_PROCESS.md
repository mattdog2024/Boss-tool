# BossTool 迭代与发布流程（强制）

> 目的：避免“聊着聊着就跑偏”，把每轮开发目标、实现、验证、发布都固化下来。

## 0. 每轮开始（输入模板）
每次新需求先用下面这句，不要省略：

```text
基于 <版本号/commit>，目标：<要实现什么>，回归标准：<怎么判定成功>，保持不变：<不能动的行为>。
```

示例：

```text
基于 v3.3，目标：修复锁屏时 Ctrl+Alt+F12 偶发失效；回归标准：锁屏/非锁屏各测试30次均可触发；保持不变：老板键 Ctrl+Alt+X 行为不变。
```

---

## 1. 开发前检查（必须）
1. `git checkout main`
2. `git pull --ff-only origin main`
3. 建议新建分支：`git checkout -b fix/<topic>`（小改可直接 main）
4. 记录当前基线：
   - `git rev-parse --short HEAD`
   - 当前 release/tag

---

## 2. 实施规范（必须）
1. 先做最小改动，不要顺手重构一堆无关代码。
2. 关键逻辑变更必须在代码里写注释（为什么改，不只是改了什么）。
3. 每个独立修复点单独 commit，禁止“全混一坨”。

推荐提交信息：

```text
fix: <问题与影响范围>
feat: <新增能力>
chore: <构建/文档/脚本>
```

---

## 3. 回归验证（必须）
至少包含：
1. **主路径测试**（新功能/修复点）
2. **反向回归**（旧功能不被破坏）
3. **边界测试**（高频触发、并发、异常状态）

BossTool 最小回归清单：
- 老板键切换（连续 30 次）
- 锁屏与解锁（密码输入、失败锁定）
- 紧急修复热键（锁屏/非锁屏都测）
- 网络切换后浏览器连通性验证

---

## 4. 版本发布（必须）
1. 合并到 `main` 并推送
2. 打 tag：`vX.Y` 或 `vX.Y.Z`
3. 创建 GitHub Release，并附带：
   - 修复列表（用户能看懂）
   - 风险/兼容性说明
   - 影响文件
4. 上传产物：
   - `BossTool_vX.Y_x64.exe`
   - `BossTool_vX.Y_x64.exe.sha256`

---

## 5. 发布后记录（必须）
更新以下文件：
- `CHANGELOG`（版本变化）
- `README`（若行为有变化）
- 本文件无需每次改，但流程变更时必须同步更新

---

## 6. 失败兜底
发布失败时按顺序排查：
1. GitHub 鉴权是否有效
2. tag 是否重复
3. release 是否已存在（存在则更新资产）
4. 构建器是否可用（MinGW 64位）

---

## 7. 不再靠“记忆”协作
以后所有迭代默认按本流程执行。  
如果你只说“继续改”，将被视为需求不完整，必须先补齐第 0 节输入模板。
